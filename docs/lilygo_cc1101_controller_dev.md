# LilyGO CC1101 Controller Developer Notes

These notes describe the LilyGO implementation for maintainers and controller
developers. End-user setup is documented in
[LilyGO T-Embed CC1101 Controller](lilygo_cc1101_controller.md).

## Architecture

The LilyGO controller is an ESPHome device built around a CC1101 radio. Home
Assistant owns the learned fireplace profile and command semantics. The firmware
provides:

- Transmit support for Proflame2 fireplace commands generated by Home Assistant.
- FIFO-based receive acquisition for guided learning.
- Active listening for packets matching the learned fireplace identity/profile.
- Diagnostic counters and status entities for troubleshooting.

The legacy edge-interval ownership path was removed after FIFO semantic receive
qualified as the active RX architecture. Do not reintroduce edge ownership unless
FIFO receive fails catastrophically and a new design is approved.

## Receive Model

The active RX path uses the CC1101 as an OOK data slicer and FIFO byte source.
Firmware captures byte windows and uses the shared Proflame2 decoder path to
identify valid candidates.

During guided learning, LilyGO sends FIFO capture events to Home Assistant. Home
Assistant promotes only validated candidates into learned profile data.

During active listening, firmware filters received candidates against the stored
learned identity/profile before publishing events to Home Assistant. This avoids
sending normal RF noise and unrelated fireplace traffic to Home Assistant.

## Active Listening

Active listening is disabled by default. When enabled:

- RX is paused during TX and resumed afterward.
- Only packets matching the learned fireplace identity/profile are published.
- Repeats from the same transmission are suppressed.
- A production dropped-packet counter is exposed.
- Additional diagnostic counters/logging may be compiled or enabled for debug
  builds.

The default scan cadence and rolling window are tuned for reliability without a
tight CPU loop. Avoid changes that increase background CPU load without
measuring the impact.

## Transmit Model

TX is timing-sensitive. The implementation deliberately keeps the critical
transmit loop compact and avoids extra abstraction inside the timing path.

Safe refactoring boundary:

- It is acceptable to split noncritical orchestration, configuration, telemetry,
  and display code.
- Do not refactor the inner TX timing loop without hardware validation.
- Any TX timing change should be verified with a real fireplace and an external
  witness such as `rtl_433`.

## ESPHome Packages

Normal production builds should include:

- `esphome/packages/proflame2_tembed_base.yaml`
- `esphome/packages/proflame2_tembed_display.yaml`

Debug-only packages and controls should remain opt-in. They are intended for
diagnostics and should not be required for normal learning, TX, or active
listening.

## Validation Expectations

After firmware changes, validate at least:

- LilyGO TX is accepted by the fireplace.
- LilyGO TX is decoded by `rtl_433` if available.
- Native remote changes are received when active listening is enabled.
- TX still works after active listening has been enabled.
- Home Assistant remains responsive and no tight active-listening retry loop is
  introduced.
