# YardStick Controller Developer Notes

These notes describe the YardStick implementation for maintainers and controller
developers. End-user setup is documented in
[YardStick One Controller](yardstick_controller.md).

## Architecture

The YardStick controller uses rfcat and a USB YardStick One. Home Assistant owns
the learned fireplace profile and command semantics. The YardStick path provides:

- Guided learning from the original remote.
- Transmit support for Proflame2 fireplace commands generated by Home Assistant.
- Diagnostic receive support used during learning and validation.

YardStick does not support active listening after setup. It should not be
presented as a state-tracking controller.

## Learning

During guided learning, Home Assistant asks the YardStick path to receive native
remote traffic and uses the shared Proflame2 candidate scanner/decoder to accept
only valid learning samples.

Accepted candidates are promoted into the same learned profile model used by the
other controllers. Avoid adding YardStick-only learned profile semantics.

## Transmit

The fireplace requires repeated frames for reliable command acceptance. The
current YardStick implementation can send one logical command as an embedded
multi-frame waveform so external decoders usually see one fireplace command
instead of several separate commands.

A software repeat fallback may still be useful for diagnostics, but it should
not be exposed as a normal end-user option unless a compatibility issue is
identified.

## USB And Runtime Constraints

YardStick reliability depends on rfcat, libusb, and host USB passthrough. When
debugging failures, distinguish integration issues from lower-level USB
visibility or passthrough issues.

Validation should include:

- YardStick TX accepted by the fireplace.
- YardStick TX decoded by `rtl_433` if available.
- Guided learning can receive original remote packets.
- Home Assistant remains responsive if the device is disconnected or
  unavailable.
