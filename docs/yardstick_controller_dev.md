# YardStick Controller Developer Notes

These notes describe the YardStick backend at the level needed to implement or
verify a compatible rfcat/YARD Stick One controller. End-user setup is
documented in [YardStick One Controller](yardstick_controller.md).

## Scope

The YardStick backend uses rfcat with a USB YARD Stick One. Home Assistant owns
profile storage, command policy, and Proflame2 packet construction. The
YardStick path is responsible for:

- Guided learning from the original remote.
- Transmitting Proflame2 commands generated by Home Assistant.
- Returning diagnostic receive data during learning/validation.

YardStick does not provide always-on active listening. Do not present it as a
state-tracking controller.

Packet layout, ECC behavior, and transmit structure are derived from
[SmartFire](https://github.com/johnellinwood/SmartFire). Use SmartFire and
[custom_components/proflame2/rf/waveform.py](../custom_components/proflame2/rf/waveform.py)
as the packet-format references.

## rfcat Radio Configuration

The backend configures rfcat/YARD Stick One as an ASK/OOK byte-window device,
matching the model used by SmartFire:

| Setting | Value |
| --- | --- |
| TX frequency | `314973000` Hz |
| RX/learning frequency | `315000000` Hz |
| Modulation | `MOD_ASK_OOK` |
| Data rate | `2400` bps |
| Sync mode | Disabled, `setMdmSyncMode(0)` when available |
| Manchester | Disabled, `setEnableMdmManchester(False)` when available |
| Packet CRC | Disabled for receive, `setEnablePktCRC(False)` when available |
| RX packet length | Fixed byte window via `makePktFLEN(...)` when available |

The implementation intentionally treats receive as a byte-window problem, not
as packet-aligned decoded bytes from the radio. Software scans candidate
bit/symbol offsets over the returned bytes and promotes only valid decoded
samples.

## Learning Receive Model

During guided learning, Home Assistant asks the YardStick backend to receive
native remote traffic. The accepted candidate is promoted into the same learned
profile model used by the other controllers. Do not add YardStick-only learned
profile semantics.

Receive failures are often lower-level rfcat/libusb/device-access failures
rather than Proflame2 decode failures. Keep those failure classes separate when
debugging.

## TX Model

The fireplace requires repeated frames for reliable command acceptance.
SmartFire used rfcat `RFxmit(..., repeat=4)` to express five transmissions, but
that repeat argument has not behaved reliably in this environment. Calling
`RFxmit(..., repeat=4)` or `repeat=5` may transmit only once.

The current backend therefore uses an internal repeat strategy:

- Default strategy: build one embedded multi-frame RF payload and call
  `RFxmit(air_payload)` once.
- Fallback strategy: call `RFxmit(...)` repeatedly in software for diagnostics
  or compatibility investigation.

The default embedded strategy aligns YardStick with LilyGO at the semantic
level: one Home Assistant command becomes one RF burst containing the repeated
frames the fireplace expects. In validation, `rtl_433` sees this as one logical
Proflame2 command and the fireplace accepts it when RF placement/power are
adequate.

Do not expose repeat strategy as a normal UI option unless a real compatibility
case requires it. It is a backend implementation detail under the common
controller interface.

## USB And Runtime Constraints

YARD Stick One reliability depends on rfcat, libusb, USB passthrough, and
physical placement. USB 3.0 ports can create interference or unreliable device
behavior; a USB extension cable can improve both RF placement and USB isolation.

The Home Assistant integration runs rfcat work outside the main event loop so a
blocking USB/rfcat call does not freeze Home Assistant. If reimplementing this
backend, preserve that isolation boundary.

## Validation Expectations

After YardStick backend changes, validate at least:

- YardStick TX is accepted by the fireplace.
- YardStick TX is decoded by `rtl_433` as one logical command when using the
  embedded repeat strategy.
- Guided learning can receive original remote packets.
- Home Assistant remains responsive when the YARD Stick is disconnected or
  unavailable.
- If changing repeat behavior, compare `rtl_433` output and real fireplace
  acceptance, not just whether rfcat returned success.
